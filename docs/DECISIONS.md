# Decisions

## 1. 단일 모놀리식 구조 유지
과제 범위에서는 도메인 경계를 패키지로 분리하고, 배포 단위는 단일 Spring Boot 애플리케이션으로 유지했다. 구현량 대비 복잡도를 줄이고 테스트 범위를 명확히 하기 위한 선택이다.

## 2. 현재 기수는 설정값으로 관리
회원용 일정 조회, 출결 처리, 출결 요약 등은 현재 기수 기준 동작이 많다. 이를 코드 상수로 박지 않고 `app.current-cohort-number`로 외부화했다.

## 3. 소프트 삭제는 상태 전이로 처리
회원 삭제는 `WITHDRAWN`, 일정 삭제는 `CANCELLED` 상태 전이로 구현했다. 기록 보존과 감사 추적을 우선한 결정이다.

## 4. 보증금 변동은 이력 중심으로 관리
잔액만 수정하지 않고 `DepositHistory`를 함께 남긴다. 패널티 차감과 환급 diff 계산을 추적 가능하게 만들기 위한 선택이다.

## 5. QR 출석 검증 순서는 서비스에 고정
명세에 적힌 검증 우선순위를 테스트로 박고 서비스에서 그대로 구현했다. 에러코드 일관성이 중요하기 때문이다.

## 6. DTO 매핑은 수동 구현
MapStruct 같은 추가 의존성 없이 record DTO와 정적 팩토리를 사용했다. 과제 범위에서 가독성과 디버깅 용이성을 우선했다.

## 7. 서비스 단위 테스트는 Mockito 중심
서비스 레이어 비즈니스 규칙 검증이 과제 핵심이므로, 저장소를 전부 mock 처리하고 시나리오 기반 단위 테스트를 작성했다.
